## 2.2 命名约定

### 2.2.1 表名命名规则

1. **全小写字母**：表名统一采用小写英文字母，单词之间以下划线（_）分隔；
2. **子系统前缀**：表名按所属子系统添加前缀标识：
   - `sys_` —— 系统管理表组（XT子系统）
   - `petition_` —— 信访办理核心表组（PT子系统）
   - `zx_` —— 专项工作表组（ZX子系统）
   - `tz_` —— 台账信息表组（TZ子系统）
   - `tj_` —— 统计预警表组（TJ子系统）
   - `qz_` —— 群众用户端表组（QZ子系统）
   - `yd_` —— 移动端配套表组（YD子系统）
3. **名词单数**：表名使用单数名词（如sys_user，不用sys_users）；
4. **关联表命名**：多对多关联表采用"主表名_从表名"模式（如sys_user_role、petition_duplicate_relation）；
5. **长度限制**：表名总长度不超过64字符。

### 2.2.2 字段名命名规则

1. **全小写字母**：字段名统一采用小写英文字母，单词之间以下划线（_）分隔；
2. **语义明确**：字段名应准确描述字段含义，避免使用a、b、c等无意义缩写；
3. **布尔类型**：以`is_`开头的字段表示布尔值（TINYINT(1)），如is_del、is_collective、is_received；
4. **日期时间**：以`_date`结尾表示DATE类型，以`_time`结尾表示DATETIME类型；
5. **外键关联**：引用其他表主键的字段命名为`{关联表名单数}_id`（如user_id、petition_id、org_id）；
6. **计数统计**：以`_count`结尾表示数量统计字段（如visitor_count、attachment_count）；
7. **冗余字段**：为提高查询效率而冗余存储的字段命名为`{源表名}_{字段名}`（如org_name冗余自sys_organization.org_name）；
8. **长度限制**：字段名总长度不超过64字符。

## 2.3 支持软件

本数据库基于MySQL 8.0 Community Edition，常用运维命令示例如下：

**（1）数据库备份命令（mysqldump）**
```sql
-- 全量备份（含存储过程、触发器和事件）
mysqldump -u root -p --single-transaction --routines --triggers --events \
  op_oil_petition > /backup/op_oil_petition_$(date +%Y%m%d).sql

-- 仅备份表结构（不含数据）
mysqldump -u root -p --no-data --single-transaction \
  op_oil_petition > /backup/op_oil_petition_schema_$(date +%Y%m%d).sql
```

**（2）数据库恢复命令（mysql）**
```sql
-- 从全量备份恢复
mysql -u root -p op_oil_petition < /backup/op_oil_petition_20260601.sql

-- 恢复到新数据库（验证用途）
mysql -u root -p -e "CREATE DATABASE op_oil_petition_restore_test \
  CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_900_ci;"
mysql -u root -p op_oil_petition_restore_test < /backup/op_oil_petition_20260601.sql
```

**（3）数据库授权命令（GRANT）**
```sql
-- 创建应用读写账号
CREATE USER 'opms_app'@'%' IDENTIFIED BY 'StrongPassword456!';
GRANT SELECT, INSERT, UPDATE, DELETE ON op_oil_petition.* TO 'opms_app'@'%';
-- 创建只读账号
CREATE USER 'opms_readonly'@'%' IDENTIFIED BY 'StrongPassword123!';
GRANT SELECT ON op_oil_petition.* TO 'opms_readonly'@'%';
-- 创建备份专用账号
CREATE USER 'opms_backup'@'localhost' IDENTIFIED BY 'StrongPassword789!';
GRANT SELECT, LOCK TABLES, SHOW VIEW, TRIGGER ON op_oil_petition.* TO 'opms_backup'@'localhost';
FLUSH PRIVILEGES;
```

**（4）数据更新示例（UPDATE）**
```sql
-- 逻辑删除（软删除）
UPDATE petition SET is_del = 1, update_time = NOW() WHERE id = 100;
-- 批量更新状态
UPDATE petition SET status = 'ARCHIVED', archive_date = NOW()
WHERE status = 'COMPLETED' AND complete_date < DATE_SUB(NOW(), INTERVAL 30 DAY);
```

## 2.4 专门说明

### 2.4.1 业务库与系统库说明

本系统采用单一数据库（op_oil_petition）方案，所有业务数据和系统管理数据存储在同一数据库中，通过表名前缀区分：

| 表前缀 | 类别 | 说明 |
|--------|------|------|
| sys_ | 系统管理表 | 用户、角色、权限、组织、字典、配置、日志等系统支撑数据 |
| petition_ | 业务核心表 | 信访事项主表及流转相关表（转交办、协办、反馈、督办等） |
| zx_ | 业务专项表 | 隐患排查、化解包案、风险评估、领导接访、特殊群体管控 |
| tz_ | 业务台账表 | 七类信访工作台账及历史数据迁移记录 |
| tj_ | 业务统计表 | 预警规则配置、预警消息记录、统计指标中间表 |
| qz_ | 群众端表 | 网上信访提交、信访评价、政策文档 |
| yd_ | 移动端表 | 消息推送记录、用户免打扰配置 |

### 2.4.2 部署环境说明

| 环境 | 服务器 | 用途 | 数据隔离 |
|------|--------|------|----------|
| 开发环境（DEV） | 学生开发机（Docker Desktop） | 编码调试、单元测试 | 独立数据库，测试数据 |
| 测试环境（TEST） | 油田测试服务器（虚拟机） | 集成测试、业务场景验证 | 独立数据库，脱敏后的仿真数据 |
| 生产环境（PROD） | 油田数据中心（物理机×5） | 正式运行 | 生产数据库，主从复制+异地备份 |

## 3.2 逻辑结构设计

本节以五大业务模块为组织单元，逐项列出各模块的数据表逻辑结构概要。完整字段定义详见附件1《各数据表字典》。

### 3.2.1 系统管理模块逻辑结构

系统管理模块包含10张数据表，支撑用户认证（对接油田统一认证OAuth2.0）、角色权限（RBAC+组织数据权限二维模型）、三级组织架构、系统字典与配置、全量操作审计日志等系统基础功能。表间关系：sys_user与sys_organization通过org_id关联，每个用户属于一个组织；sys_user与sys_role通过sys_user_role实现多对多关联；sys_dict_type与sys_dict_item通过dict_code实现一对多关联；三张日志表（sys_login_log、sys_operation_log、sys_api_call_log）通过user_id与sys_user关联。

### 3.2.2 信访办理模块逻辑结构

信访办理模块包含9张数据表，是系统的核心业务模块。以petition（信访事项主表）为枢纽：petition与petition_dispatch通过petition_id实现一对多关联，一件信访件可产生多次派单（退回后重派）；petition_dispatch与petition_assist_record通过dispatch_id实现一对多关联（仅转办类型）；petition与petition_process_record通过petition_id实现一对多关联，支持分阶段多次添加办理记录（按时间倒序展示为时间线）；petition与petition_final_feedback通过petition_id实现一对多关联（退回修改后重新提交产生多条记录）；petition与petition_supervision通过petition_id实现一对多关联；petition与petition_duplicate_relation通过petition_id实现一对多关联；各业务表均可通过business_type+business_id与attachment关联。

### 3.2.3 专项工作模块逻辑结构

专项工作模块包含6张数据表，各表独立运作，均通过组织ID与sys_organization关联，通过登记人/创建人ID与sys_user关联。zx_risk_screening使用PC+年份+流水编号；zx_key_resolution使用HJ+年份+流水编号，可关联多个信访件；zx_stability_risk_assessment使用PG+年份+流水编号，四项评分自动判定风险等级；zx_leader_reception使用JF+年份+流水编号；zx_special_group_control独立管理特殊群体信息；zx_business_template提供标准化文档模板。

### 3.2.4 台账信息模块逻辑结构

台账信息模块包含8张数据表，每类台账独立字段模板，通过register_by与sys_user关联。tz_historical_migration_record记录对petition表的历史数据批量导入情况。所有台账表支持新增、编辑（留痕）、逻辑删除（回收站可恢复）、附件管理、Excel导出功能。

### 3.2.5 统计预警与公共服务模块逻辑结构

本模块包含8张数据表：tj_alert_rule_config定义规则，tj_alert_message_record记录每次规则触发的预警实例；tj_statistics_indicator为独立的数据仓库中间表（星型模型），数据通过ETL从业务表汇聚；qz_online_petition通过petition_id与petition关联（审核通过后转入PT流程）；qz_petition_evaluation通过petition_id与petition关联；yd_message_push_record通过target_user_id与sys_user关联。

## 3.3 物理结构设计

### 3.3.1 索引设计规范

本系统索引设计遵循以下8条原则：

| 原则 | 内容 | 说明 |
|------|------|------|
| **原则1** | 所有主键（PRIMARY KEY）自动建立聚簇索引 | InnoDB引擎以主键为聚簇索引，数据按主键顺序物理存储 |
| **原则2** | 所有唯一约束字段（UNIQUE KEY）建立唯一索引 | 如petition_code、supervision_code、username等业务编码和账号字段 |
| **原则3** | 所有WHERE子句高频查询字段建立普通索引 | 如status、channel_code、org_id、register_time等高频筛选字段 |
| **原则4** | 所有关联外键字段建立普通索引 | 如petition_id、user_id、dispatch_id等JOIN关联字段，不创建物理外键约束 |
| **原则5** | 大文本字段（TEXT）不建立索引 | 诉求内容、反馈内容等大字段通过Elasticsearch实现全文检索 |
| **原则6** | 低基数字段不单独建立索引 | 如is_del（仅0/1）、is_collective（0/1），但可作为复合索引的前缀 |
| **原则7** | 频繁更新的字段不建立索引 | 如update_time作为ON UPDATE CURRENT_TIMESTAMP频繁变化，维护开销大 |
| **原则8** | 考虑建立覆盖索引以减少回表 | 对高频查询的字段组合（如status+register_time）建立复合索引 |

**索引命名规则**：

| 索引类型 | 命名格式 | 示例 |
|----------|----------|------|
| 主键索引 | PRIMARY KEY | PRIMARY KEY (id) |
| 唯一索引 | uk_{字段名} | uk_petition_code |
| 普通索引 | idx_{字段名} | idx_status |
| 复合索引 | idx_{字段1}_{字段2} | idx_business (business_type, business_id) |

### 3.3.2 数据存放方案

**（一）数据与索引分磁盘存放**

数据库服务器配置RAID10磁盘阵列（4×8TB HDD），高并发大表可通过MySQL的`DATA DIRECTORY`和`INDEX DIRECTORY`表选项分离存储：

```sql
CREATE TABLE tj_statistics_indicator (
    ...
) ENGINE=InnoDB
DATA DIRECTORY = '/data/mysql_data/'
INDEX DIRECTORY = '/data/mysql_index/';
```

**（二）大表拆分策略**

| 策略 | 适用表 | 实现方式 |
|------|--------|----------|
| 按月分区 | sys_login_log、sys_operation_log | RANGE PARTITION BY TO_DAYS(time_column) |
| 冷热数据分离 | petition（历史归档件） | 超过3年的已归档数据迁移至历史表 |
| 垂直拆分 | petition（敏感字段） | 身份证号/手机号等加密字段可独立为子表 |

**（三）备份异地存储**

数据库备份文件经GPG加密后，通过油田内网光纤同步至B机房备份服务器（AMD EPYC 7232P / 32GB ECC / 4×8TB HDD RAID5 可用24TB），RPO=24小时，RTO=4小时。

1. **完整性**：每张表的所有字段必须在数据字典中逐项列出，不得遗漏；
2. **一致性**：字段名、数据类型、约束条件须与DDL脚本保持严格一致；
3. **可读性**：字段说明须使用中文描述，准确表达业务含义，非技术人员可理解；
4. **规范性**：拼音代码按照中文拼音首字母缩写法编制，遵循国标GB/T 1526惯例；
5. **维护性**：数据字典随数据库变更同步更新，版本号与DDL脚本版本号一一对应。

# 5 其它设计

## 5.1 完整性设计

### 5.1.1 实体完整性

所有数据表均通过主键（PRIMARY KEY）确保实体完整性：主键均为自增BIGINT（id），由MySQL自动生成唯一值；业务编码字段（petition_code、supervision_code、assessment_code等）通过UNIQUE KEY确保业务唯一性。

### 5.1.2 参照完整性

本系统**不创建物理外键约束**（FOREIGN KEY），参照完整性由业务代码层保证，理由如下：

1. **性能考虑**：物理外键在INSERT/UPDATE/DELETE时需检查约束，在高并发写入场景下影响性能；
2. **灵活性考虑**：信访业务中存在逻辑删除（is_del=1）和归档迁移等特殊操作，物理外键的级联规则难以覆盖全部业务场景；
3. **运维考虑**：数据迁移、批量导入时，物理外键约束会导致导入失败或需要按依赖顺序导入，增加运维复杂度。

业务代码层通过以下机制保障参照完整性：ORM框架（MyBatis-Plus）在Service层进行关联数据存在性校验；定期数据完整性巡检脚本检查孤立记录并生成报告；关键关联字段（如petition_id、user_id、org_id）均建立普通索引加速校验查询。

### 5.1.3 用户定义完整性

| 约束类型 | 实现方式 | 示例 |
|----------|----------|------|
| 非空约束 | NOT NULL | petition_code、petitioner_name、channel_code |
| 唯一约束 | UNIQUE KEY | username、employee_no、screening_code |
| 默认值 | DEFAULT | is_del DEFAULT 0、status DEFAULT 'DRAFT' |
| 检查约束 | 应用层@Valid校验 | score_legality 1-10范围、total_score = 四项之和 |
| 外键引用 | 应用层存在性校验 | petition_id须存在于petition表 |

## 5.2 安全保密设计

### 5.2.1 身份认证与分级权限管控

1. **身份认证**：系统对接油田统一身份认证系统（OAuth2.0），用户密码由统一认证系统管理，本系统不存储明文密码；
2. **分级权限**：采用"RBAC角色+组织数据权限"二维模型——四类角色分配不同功能权限，七条数据权限规则控制数据可见范围；
3. **数据库账号分级**：

| 数据库账号 | 权限范围 | 使用场景 |
|------------|----------|----------|
| opms_app | SELECT, INSERT, UPDATE, DELETE | 应用服务日常业务操作 |
| opms_readonly | SELECT | 统计查询、报表生成 |
| opms_backup | SELECT, LOCK TABLES, SHOW VIEW, TRIGGER | 数据库备份 |

### 5.2.2 四类故障处理

| 故障类型 | 故障描述 | 应对方案 |
|----------|----------|----------|
| **系统故障** | 操作系统崩溃、服务器宕机 | MySQL InnoDB通过redo log自动恢复已提交事务；双机热备自动切换 |
| **介质故障** | 硬盘损坏、RAID降级 | RAID10允许最多2盘故障；故障盘热插拔更换后自动重建 |
| **事务故障** | 事务执行中断、死锁 | InnoDB自动检测死锁并回滚代价较小的事务；应用层事务超时30秒 |
| **用户错误** | 误删除、误更新数据 | 逻辑删除（is_del=1）使删除可逆；通过每日备份恢复个别记录 |

### 5.2.3 逻辑与物理双备份方案

| 备份层次 | 备份方式 | 说明 |
|----------|----------|------|
| **逻辑备份** | mysqldump每日全量 | 每日凌晨2:00执行，GPG加密后同步至B机房 |
| **逻辑备份** | binlog增量 | 持续记录所有变更，配合全量备份实现任意时间点恢复 |
| **物理备份** | 附件文件rsync | 每日同步NAS中新增/变更的附件文件至备份服务器 |
| **物理保护** | RAID磁盘冗余 | 数据库服务器RAID10（4盘）、备份服务器RAID5（4盘） |

## 5.3 备份机制

### 5.3.1 备份策略

| 备份类型 | 执行频率 | 执行时间 | 保留策略 | 存储位置 |
|----------|----------|----------|----------|----------|
| 每日全量备份 | 每天1次 | 凌晨2:00 | 保留最近30天 | B机房备份服务器 |
| 每月归档备份 | 每月1次 | 每月1日凌晨3:00 | 保留最近12个月 | B机房备份服务器 |
| binlog增量备份 | 实时（MySQL自动） | 持续 | 保留最近30天 | 本地+B机房同步 |
| 附件增量同步 | 每天1次 | 凌晨2:30 | 随全量备份保留 | B机房备份服务器 |

### 5.3.2 故障恢复方案

**场景一：单表数据恢复**
```sql
mysql -u root -p op_oil_petition < /restore/single_table_petition.sql
```

**场景二：数据库完整恢复**
```bash
mysql -u root -p op_oil_petition < /backup/op_oil_petition_20260606.sql
mysqlbinlog --start-datetime="2026-06-06 02:00:00" /var/log/mysql/mysql-bin.* | mysql -u root -p op_oil_petition
```

**场景三：异地容灾恢复（B机房数据完好，A机房全部损毁）**
```bash
# RPO=24小时，RTO=4小时
mysql -u root -p < /backup_b/op_oil_petition_20260605.sql
rsync -av /backup_b/attachments/ /nas/oilfield/petition/
```

### 5.3.3 备份验证

每季度执行一次备份恢复演练：在独立测试环境中从备份文件恢复数据库；校验核心表（petition、sys_user）记录数；抽查10条信访件附件完整性；记录演练结果于《数据库备份恢复演练报告》。
