<a id="11-编写目的"></a>
## **1.1 编写目的**

本文档旨在对油田信访管理系统进行完整、详尽的软件需求分析，明确系统的功能边界、业务规则、性能指标及约束条件，为后续的系统设计、开发、测试及验收提供统一的依据和标准。

本文档的预期读者包括：

<a id="12-项目背景"></a>
## **1.2 项目背景**

### **1.2.1 油田信访工作的现实挑战**

新疆油田作为我国西部最大的油气生产基地，勘探开发区域集中在准噶尔盆地，面积约13万平方千米，基层站点分布在广袤的戈壁和山地之间。油田现有在职员工近3万人，离退休人员3万余人，具有典型的"点多、线长、面广"行业特征。这种布局决定了信访工作天然面临着"人散、事远、协调难"的挑战——一位在玛湖油田工作的采油工，如果要到克拉玛依市区的油田总部信访窗口反映问题，往返至少需要半天时间。

近年来，随着油田深化改革的持续推进——包括"三项制度"改革（劳动、人事、分配）、"油公司"模式改革、矿区"三供一业"分离移交等——员工的利益格局正在经历深刻重构。员工诉求从过去相对单一的"工资待遇"问题，扩展为涵盖劳动用工、薪酬福利、社会保障、住房安置、历史遗留、环境保护等多个领域的复杂矩阵。

当前油田信访工作面临以下六大突出挑战：

1. **渠道多头管理难**：来信、来访、来电、网上信访、上级转送、领导批示等多渠道信息分散，缺乏统一受理入口，容易造成遗漏或重复办理。
2. **流程断点协同难**：信访事项涉及多层级、多部门，传统纸质流转或线下沟通方式效率低下，转办路径不清晰，主办协办责任界定模糊。
3. **办理时效管控难**：缺乏自动化的时限预警和催办机制，超期办理时有发生，督办依赖人工记忆和经验，缺乏刚性约束。
4. **数据沉淀研判难**：大量历史信访数据分散在各类台账和报表中，无法有效整合分析，风险预警依赖人工经验，缺乏数据驱动的智能研判能力。
5. **事前防控落地难**：不稳定因素排查、重点事项化解、稳定风险评估等专项工作缺乏统一的线上管理工具，信息流转慢、跟踪不到位。
6. **移动办公支撑难**：信访工作人员和基层专项人员经常需要深入一线、走访现场，待办事项、预警信息无法及时触达，现场采集的排查信息无法即时回传。

### **1.2.2 政策法规驱动**

2022年5月1日施行的新版《信访工作条例》第二十一条，以行政法规的形式明确要求："各级党委和政府应当加强信访工作信息化、智能化建设，依规依法有序推进信访信息系统互联互通、信息共享"，并进一步要求"各级机关、单位应当及时将信访事项录入信访信息系统，使网上信访、来信、来访、来电在网上流转，方便信访人查询、评价信访事项办理情况"。这意味着信息化、智能化不再只是技术升级的选择题，而是合规运营的必答题。

国家信访局配套印发的《信访事项网上办理工作规程》（国信发〔2022〕8号）对信访事项的登记、受理、办理、送达等各环节提出了详细的操作规范，明确要求：信访事项应在15日内完成转送和处理分类，受理后60日内办结（情况复杂可延长不超过30日），办理过程和结果须在网上向信访人反馈，实现"可查询、可跟踪、可监督、可评价"。

### **1.2.3 行业先行实践**

国内兄弟油田企业已在信访管理数字化领域进行了积极探索：

- **河南油田**：2024年9月上线"信访矛盾调解管理系统"+ 手机端小程序"遇有事·依法调"，构建三级网格体系（1个一级网格、16个二级网格、192个基础网格、1600余名网格员），覆盖3万余人，运行后工作效率提升90%。
- **中原油田**：建设数智化调解平台，整合维稳、人力、社保、档案等10余个部门资源，创新引入DeepSeek大模型提供智能咨询，网上信访平台诉求处结率100%，累计办理业务4.7万人次。
- **江汉油田**：2025年初上线"码上就办"员工诉求平台，分设五大板块，一个月收集诉求130余项，响应率100%。
- **长庆油田采油五厂**：2025年4月上线"有事您说话"扫码平台，办结率100%，平均响应时间仅3.8天。

上述实践表明，油田企业信访管理的数字化转型不仅技术上成熟可行，而且成效显著。

### **1.2.4 新疆油田的现实差距**

与上述兄弟油田相比，新疆油田目前的信访管理工作仍在较大程度上依赖传统手工模式：信访登记仍以人工手写登记表+事后录入Excel为主；信访件流转依赖纸质函件或电话沟通，缺乏在线流转和环节留痕；判重依赖人工记忆和经验，准确率不稳定；各单位之间的信访信息各自为政，形成"信息孤岛"；催办督办依赖人工跟踪和电话提醒；统计报表每月手工汇总Excel，耗时且容易出错；缺乏移动端功能。这些差距直接导致了办理效率不高、数据利用率低、决策缺乏量化支撑等痛点。

<a id="13-建设目标"></a>
## **1.3 建设目标**

本系统建设的总体目标是：构建以"多渠道受理、全流程闭环、多层级协同、智能化研判"为核心的油田信访管理系统，实现信访事项从登记、判重、分类处置、转交办、办理反馈、督办催办到办结归档的全链条线上化与智能化闭环管控。

具体建设目标分为六大维度：

**（一）全流程闭环管理**：实现信访事项全生命周期线上流转，各环节强制留痕，办理路径可追溯，责任主体可落实。信访事项全程线上流转率 ≥ 95%。

**（二）多层级协同联动**：支撑油田公司、厂处、作业区三级业务流转，明确交办（向下）、转办（平级）、协办（配合）、呈报（向上）四种流转模式的业务规则和权限边界。

**（三）智能化提质增效**：通过智能判重、智能分类推荐、自动时限预警与催办、风险智能预警等能力，将事务性工作交由系统自动完成，人工聚焦于审核、决策和例外处理。目标将信访事项平均办理周期从当前约25天缩短至15天以内。

**（四）数据化研判预警**：汇聚全业务数据，提供多维度统计分析和风险智能预警，辅助管理层掌握态势、科学决策。重复信访率降低20%以上。

**（五）全要素台账整合**：将文件制度、报告函件、包案管理、信访档案、业务培训、人员信息及历史资料等统一归集，消除信息孤岛。

**（六）移动化即时响应**：通过移动端H5页面（集成至"掌上新油"移动办公平台），为信访管理员、业务办理人、专项工作人员提供轻量化移动工作台，实现待办消息实时推送、关键业务移动办理、现场信息即时采集。

**核心量化指标**：

| 指标项 | 目标值 |
|--------|--------|
| 信访事项全程线上流转率 | ≥ 95% |
| 平均办理周期 | ≤ 15天 |
| 重复信访率降低 | ≥ 20% |
| 信访人满意度 | ≥ 90% |
| 办结率 | ≥ 98% |
| 系统年度可用率 | ≥ 99.5% |

<a id="14-适用范围"></a>
## **1.4 适用范围**

本系统面向新疆油田公司全体系信访业务，覆盖范围如下：

**组织覆盖**：油田公司总部（1个）、二级单位（约23个，含采油厂、研究院、工程技术服务单位等）、基层站队/作业区（约120个）。

**用户覆盖**：系统注册用户约500人，涵盖信访管理员（约80人）、业务办理人（约420人）、专项工作人员、系统管理员及信访群众五类角色。

**业务覆盖**：
- 信访事项的全流程办理与闭环管理（含网上信访群众服务）；
- 不稳定因素排查、重点事项化解、稳定风险评估、领导接访下访、特殊群体管理等专项工作；
- 信访数据的统计分析与风险智能预警；
- 七类台账的管理与历史数据迁移。

**终端覆盖**：PC端覆盖全量功能（通过油田内部局域网访问），移动端覆盖高频操作场景（通过"掌上新油"平台H5页面访问）。

<a id="15-术语定义"></a>
## **1.5 术语定义**

| 术语 | 英文缩写 | 定义 |
|------|----------|------|
| 信访件 | Petition | 信访人通过来信、来访、来电、网络等渠道反映诉求的单个事项记录，系统为其分配唯一信访编号 |
| 判重 | Dedup | 系统自动或人工判断新登记信访件与历史信访件是否为同一人或同一事项的重复反映 |
| 交办 | Assign-Down | 上级信访部门将信访事项交由下级单位办理，接收方为主办责任方 |
| 转办 | Transfer-Level | 信访部门将信访事项转给同级业务部门办理，接收方为主办责任方 |
| 协办 | Assist | 在主办部门牵头下，其他相关部门配合完成特定协助事项 |
| 呈报 | Report-Up | 信访部门将重大信访事项上报给上级领导或上级信访部门批示 |
| 主办部门 | Lead Dept | 对信访事项办理结果承担第一责任的部门，每个信访件有且仅有一个主办部门 |
| 协办部门 | Assist Dept | 在主办部门协调下提供情况、数据或专业意见的配合部门，可多个 |
| 督办 | Supervision | 信访部门对办理滞后或落实不力的单位或部门发出限期整改通知的业务行为 |
| 掌上新油 | ZSXY | 油田企业移动办公平台，本系统通过该平台实现消息推送与移动端H5页面访问 |
| 不稳定因素排查 | Risk Screening | 各单位定期对可能引发信访或群体性事件的隐患进行摸排登记的业务工作 |
| 稳定风险评估 | Stability Assessment | 对涉及员工切身利益的重大决策、项目实施前进行风险识别和等级评定的工作 |
| 包案管理 | Case Assignment | 由领导干部负责某一重点信访事项的化解，实行"一包到底"的责任制管理 |
| 信访群众用户端 | Public Portal | 面向油田员工及离退休人员的网上信访自助服务平台 |

# **二、项目概述**

<a id="21-系统概述"></a>
## **2.1 系统概述**

本系统是面向新疆油田企业的数字化信访维稳管理平台，核心能力为"多渠道受理、全流程闭环、多层级协同、智能化研判"。系统覆盖油田公司、厂处、作业区三级单位的信访办理、专项工作、统计分析及台账管理业务，采用Spring Boot + Vue.js前后端分离架构，MySQL 8.0数据库（主从复制），Docker Compose容器化部署。

系统由七大功能子系统构成，共计47个一级功能模块：

| 子系统 | 编号 | 模块数 | 核心定位 |
|--------|------|--------|----------|
| 信访办理子系统 | PT | 10 | 核心业务流程，覆盖信访件全生命周期 |
| 专项工作子系统 | ZX | 6 | 信访维稳延伸业务场景 |
| 统计分析子系统 | TJ | 4 | 数据化研判与决策支持 |
| 台账信息子系统 | TZ | 2 | 七类台账管理与历史数据迁移 |
| 移动端子模块 | YD | 7 | 轻量化移动工作台 |
| 系统管理子系统 | XT | 6 | 基础配置与系统维护 |
| 信访群众用户端 | QZ | 6 | 面向员工群众的网上信访自助服务 |
| **特殊：全局通用功能** | GL | 4 | 消息推送、权限管理、数据导出、日志审计 |

<a id="22-角色分析"></a>
## **2.2 角色分析**

系统面向五类用户，通过智能化手段接管判重、分类、催办、统计等事务性工作，人工聚焦于审核、决策和例外处理。

| 角色 | 核心定位 | 主要职责 | 智能化替代的人工操作 | 使用端 |
|------|----------|----------|----------------------|--------|
| 信访管理员 | 中枢调度与审核中心 | 确认复杂关联件、审核办理结果、发起重大事项督办、维护智能化规则与业务模板、查看统计分析与预警 | 免去手动判重、手动分类、手动催办提醒、手动生成报表 | PC管理端（全量功能） |
| 业务办理人 | 执行与反馈单元 | 签收任务、填写办理情况与反馈、申请延期或退件、特殊情况下手动发起协办、审核本部门办理结果 | 免去手动签收、手动计算时限、常规协办请求自动流转 | PC管理端（办理相关功能） |
| 专项工作人员 | 基层信息采集与预警终端 | 核实并填报系统筛选出的风险信息、落实稳控措施并更新动态、排查与评估数据的录入与导入 | 免去大海捞针式排查、手动预警；系统自动筛选高风险目标并生成排查建议任务 | PC端 + 移动端（现场采集） |
| 系统管理员 | 规则与系统维护者 | 用户与权限配置、组织架构同步、业务字典维护、模板上传与版本管理、日志审计与系统监控 | 为智能化运行提供基础配置保障 | PC管理端 |
| 信访群众 | 诉求提出与进度跟踪 | 通过网上信访大厅提交诉求、查看办理进度、评价办理结果、查阅政策法规 | 从"跑腿信访"变为"指尖信访" | Web用户端 + "掌上新油"移动端 |

<a id="23-组织结构与业务流转"></a>
## **2.3 组织结构与业务流转**

| 组织层级 | 在信访业务中的核心职能 | 关键业务动作 | 数据查看范围 |
|----------|------------------------|--------------|--------------|
| 油田公司（一级） | 统筹管理、监督指导、重大事项决策 | 向下级单位交办、向同级职能部门转办、向公司领导呈报重大事项、对全公司信访件发起督办、制定与发布业务模板 | 全公司信访与专项工作数据 |
| 厂处级（二级） | 属地管理、承上启下、办理主体 | 承接公司交办任务并反馈、向作业区交办、向同级职能部门转办、向公司呈报或申请退件、对本级及下级发起督办、组织不稳定因素排查 | 本厂处及下属作业区数据 |
| 作业区级（三级） | 基层执行、前端感知、稳控落实 | 承接厂处交办任务并反馈、开展不稳定因素排查上报、落实重点人员稳控措施并更新动态、向厂处呈报重大隐患 | 本作业区数据 |

**四种流转模式**：

| 流转动作 | 发起方 | 接收方 | 方向 | 责任性质 |
|----------|--------|--------|------|----------|
| 交办 | 上级信访部门 | 下级单位/部门 | 向下 | 下级单位为主办责任方，必须按期反馈结果 |
| 转办 | 信访部门 | 同级业务部门 | 平级 | 接收部门为主办部门，对办理结果负总责 |
| 协办 | 主办部门或信访部门 | 其他相关部门 | 可跨同级或向下 | 协办部门承担配合责任，需按时回复 |
| 呈报 | 信访部门 | 上级领导或上级信访部门 | 向上 | 呈报对象作出批示或决策 |

<a id="24-智能化程度约定"></a>
## **2.4 智能化程度约定**

| 业务场景 | 智能化实现方式 | 人工介入程度 |
|----------|----------------|--------------|
| 信访件判重 | 系统自动比对历史库，基于姓名+身份证号精确匹配、姓名+电话+关键词模糊匹配、员工编号关联匹配三级规则。相似度超阈值自动关联，阈值区间内推管理员确认 | 仅复杂关联件需人工确认 |
| 诉求分类与分流建议 | 系统根据诉求内容自动推荐三级分类及转交办建议（基于可配置规则表） | 管理员一键确认或修改 |
| 办理催办 | 系统根据时限规则自动向办理人推送催办消息，在时限过半（50%）、剩余3天、剩余1天、超期当日及之后每3天五个节点自动触发 | 无需人工介入 |
| 风险预警 | 系统实时扫描数据流，触发五类预警规则后自动生成预警并推送至责任人 | 人工处置预警，标记处理状态 |
| 不稳定因素排查 | 系统结合多源数据自动筛选高风险目标，生成排查建议任务 | 专项工作人员核实填报 |
| 统计报表 | 标准化报表由系统自动计算填充数据，支持同比环比和三级下钻 | 人工一键导出或查看 |
| 模板管理 | 信访管理员上传维护，系统自动版本控制，各单位下载使用最新版 | 仅模板内容变更时人工更新 |

<a id="25-假定和约束"></a>
## **2.5 假定和约束**

### **2.5.1 假定**

本系统的建设基于以下基本假定：

1. 油田公司、厂处、作业区的组织架构及隶属关系已在人力资源系统中明确，系统可通过接口同步获取。
2. 各级单位已具备油田内网接入条件，"掌上新油"移动平台已在全油田范围内推广使用。
3. 油田已建有统一的身份认证系统（LDAP/AD域或统一认证平台），系统可与其对接实现单点登录（SSO）。
4. 信访办公室将按时提供不稳定因素排查、稳定风险评估等业务的标准模板（Excel格式）。
5. 历史信访数据以Excel表格或旧系统数据库形式存在（约10万条记录），可进行清洗和迁移。
6. 用户具备基本的计算机操作能力，经过短期培训（约2小时）即可熟练使用系统。
7. 油田信息技术部可提供服务器资源和网络环境支持。

### **2.5.2 约束**

本系统的设计、开发与部署需遵守以下约束条件：

1. **技术约束**：必须遵循油田公司信息化建设相关技术标准和安全规范。
2. **认证约束**：必须与油田统一身份认证系统对接，不单独建立用户账号体系。
3. **平台约束**：必须与"掌上新油"移动平台对接，实现消息推送和移动端H5页面嵌入。
4. **部署约束**：系统需部署在油田内网环境，网络安全等级保护不低于二级（等保2.0）。
5. **隐私约束**：信访人的个人隐私信息（身份证号、电话号码、住址）需AES-256加密存储，展示时按角色进行脱敏处理。
6. **迁移约束**：历史数据迁移需保证数据完整性，迁移前后数据记录数及核心字段需可校验。
7. **工期约束**：项目建设期为1个月（28个日历日），核心信访办理流程须在第一期上线。
8. **预算约束**：项目总投资约83万元（校企合作口径）。

<a id="26-系统边界"></a>
## **2.6 系统边界**

本系统覆盖油田信访工作的核心业务，但以下内容不在本期建设范围内：

1. 移动端原生App开发（iOS/Android），仅通过H5页面集成至"掌上新油"实现移动端功能。
2. 信访件办理过程中的视频会议、即时通讯等实时协同功能。
3. 与油田外部政府部门信访系统的数据自动交换，仅支持手动导出导入。
4. 信访人的生物特征识别（人脸、指纹）与身份验证。
5. 旧系统历史数据的清洗和整理工作（由信访办公室负责），系统仅提供导入工具和校验功能。
6. AI大模型智能问答（如DeepSeek），预留接口供后续迭代。

<a id="27-pc端与移动端需求划分"></a>
## **2.7 PC端与移动端需求划分**

### **2.7.1 PC端承载功能**

PC端作为系统的核心操作平台，承载全量功能（41个管理端模块），面向信访管理员、业务办理人、专项工作人员和系统管理员四类内部角色。

主要业务板块：
- **信访办理**：信访件从登记到归档的全生命周期管理，含批量操作、时限自动监控、流程节点流转和全过程留痕。
- **专项工作**：不稳定因素排查、重点事项化解、稳定风险评估、领导接访下访、特殊群体管理、模板与资料维护。
- **统计分析**：访量趋势统计、人员诉求统计、标准月报/年报自动生成、风险智能预警。
- **台账管理**：七类台账的新增、编辑、查询、导出及历史数据迁移。
- **系统管理**：用户与权限配置、组织架构同步、业务字典维护、模板版本管理、日志审计。

### **2.7.2 移动端承载功能**

移动端定位为PC端信访管理系统的轻量化延伸工具，以H5页面形式集成至油田移动办公平台"掌上新油"，聚焦"离开工位也能办公"的核心诉求，围绕待办处理、消息接收、现场信息采集三大场景设计，不做PC端全量功能的搬运。

**移动端包含功能**：
- 待办中心（按紧急程度排序、类型筛选）
- 信访件处理（签收、反馈、拍照上传、申请延期）
- 领导审批（查看呈报摘要、快捷审批、手写签名）
- 现场信息采集（排查信息录入、现场拍照、重点人员动态更新、一键上报）
- 消息通知（待办提醒、催办通知、预警推送、历史消息列表）
- 信息查询（按编号/姓名检索、核心指标卡片查看）
- 个人设置（免打扰时段、修改密码、退出登录）

**移动端明确排除的功能**：信访件完整登记、转交办配置（多级选人交互复杂）、判重确认（需仔细比对）、多维统计图表、台账管理、系统管理。

<a id="三功能性需求"></a>
# **三、功能性需求**

<a id="31-信访办理子系统pt"></a>
## **3.1 信访办理子系统（PT）**

信访办理子系统是本系统的核心业务模块，覆盖信访件从多渠道登记、判重、分类处置、转交办、办理反馈、督办到办结归档的完整生命周期。以下按10个模块逐一详述。

<a id="311-待办管理pt-01"></a>
### **3.1.1 待办管理（PT-01）**

#### **业务背景**

待办管理是用户登录系统后的首要落地页面。在传统工作模式下，信访工作人员需要从纸质文件堆、邮件收件箱、电话记录中逐一梳理今天要处理的事项——哪些新信访件需要登记、哪些交办件需要签收、哪些快超期了需要催办。待办管理将这些分散的信息统一聚合，以任务列表形式清晰展示。

#### **功能详述**

**（1）待办任务列表**

系统需将当前用户所有待处理任务按类型分组展示，任务类型包括：
- 待登记：从网上信访用户端提交的信访件，等待信访管理员补全信息和正式登记（来源：QZ-01提交的简化式信访）
- 待签收：上级交办或平级转办的信访件，等待承办单位签收确认
- 待反馈：已签收正在办理中的信访件，等待提交办理结果
- 待审核：承办单位已提交反馈的信访件，等待信访科审核（内部审核通过后的外部审核）
- 待归档：已审核通过等待确认办结归档的信访件
- 待处理预警：系统自动生成的预警信息，等待责任人处置

每条任务展示以下信息：
- 事项编号（信访编号/督办编号/预警编号）
- 信访人姓名（脱敏展示，如"张**"）
- 诉求类别及简要描述（不超过50字）
- 当前环节说明（待签收/办理中/即将超期/已超期）
- 剩余时限或超期天数
- 任务到达时间

**（2）紧急程度标识**

根据办理时限剩余天数，用不同颜色区分紧急程度：

| 状态 | 颜色 | 触发条件 |
|------|------|----------|
| 正常 | 绿色 | 剩余时限 > 7天 |
| 临近超期 | 橙色 | 剩余时限 ≤ 7天且 > 0天 |
| 紧急 | 红色 | 剩余时限 ≤ 3天且 > 0天 |
| 已超期 | 深红色加粗 | 剩余时限 ≤ 0天 |

超期任务自动置顶显示，在同一紧急程度内按超期天数降序排列。

**（3）筛选与排序**

支持以下筛选条件：
- 任务类型（多选）：待签收、待反馈、待审核、待归档
- 来源渠道：来信、来访、来电、网上信访、上级转送、领导批示
- 时限范围：今日到期、3日内到期、7日内到期、已超期
- 关键字搜索：按信访编号或信访人姓名模糊搜索

支持以下排序方式：
- 按紧急程度（默认，超期优先）
- 按交办/到达时间
- 按剩余时限

**（4）批量操作**

同一环节的任务支持批量操作：
- 批量签收：勾选多条待签收任务，一键批量签收
- 批量转办：勾选多条待转办任务，一键批量转办至同一部门

批量操作前需二次确认，操作结果以列表形式反馈（成功N条，失败M条及失败原因）。

**（5）待办统计卡片**

页面顶部展示当前用户的待办统计：

| 统计项 | 说明 |
|--------|------|
| 待签收数 | 等待签收的任务数量 |
| 即将超期数 | 剩余时限 ≤ 3天的任务数量 |
| 已超期数 | 已超过办理时限的任务数量 |
| 待办总数 | 所有待处理任务的合计数量 |

点击任一统计数字，列表自动筛选至对应类型的任务。

**（6）实时刷新**

- 待办列表每60秒自动增量刷新
- 当收到新的催办消息或任务分配时，列表强制刷新
- 用户可手动点击刷新按钮即时刷新

#### **业务规则**

1. 各角色看到的待办任务仅限本人权限范围内的数据。
2. 信访管理员可查看全公司的待办统计数据；业务办理人仅可查看本部门的待办任务。
3. 已超期任务的超期天数从超期当日开始累加计算。
4. 批量操作中任一任务操作失败不影响其他任务的操作结果。

<a id="312-多渠道登记pt-02"></a>
### **3.1.2 多渠道登记（PT-02）**

#### **业务背景**

《信访工作条例》第二十一条和《信访事项网上办理工作规程》第四条明确要求：通过信息网络、书信、电话、传真、走访等所有形式提出的信访事项，均应客观、准确、及时登记录入信访信息系统。统一登记入口的设计消除了渠道间的数据标准差异，确保所有信访件同质化管理。

#### **功能详述**

**（1）统一登记入口**

无论是来信、来访、来电、网上信访、上级转送件还是领导批示件，均通过统一表单登记。登记员首先选择信访来源渠道（下拉选择），系统根据选择动态展示该渠道的差异化字段，同时自动生成信访编号。

**信访编号编码规则**：
- 格式：`XF` + `YYYYMM` + `4位流水号` + `2位渠道标识`
- 示例：XF2026050001LF（2026年5月第1号，来访）
- 渠道标识：LF=来访、LX=来信、LD=来电、WS=网上信访、SZ=上级转送、LP=领导批示
- 流水号：每月从0001开始，按月重置

**（2）通用字段（所有渠道必填或按需填写）**

**信访人信息**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| 姓名 | 文本(50) | 是 | 信访人真实姓名 |
| 身份证号 | 文本(18) | 否 | 支持15位旧号，系统校验校验位 |
| 员工编号 | 文本(20) | 否 | 若为本企业员工，选择后自动带出HR系统信息 |
| 联系电话 | 文本(11) | 是 | 手机号或座机号 |

- **历史提示**：输入信访人姓名+联系电话后，系统实时提示"该信访人曾于X年X月X日反映过XX问题（信访编号：XXX），当前状态：XXX"，帮助登记员快速了解信访人历史。
- **关键字自动提取**：输入诉求内容后，系统基于IK Analyzer中文分词自动提取高频关键词，推荐为标签。
- **必填校验**：提交时校验所有必填字段，未填项标红提示。

**（5）登记提交流程**

1. 登记员填写/确认所有必填信息
2. 点击"提交"按钮

4. 若判重结果为"疑似重复"，弹出判重提醒窗口（详见PT-03）
5. 若判重结果为"非重复"或登记员确认为新件，信访件保存并进入待分类处置状态
6. 系统自动记录登记日志（登记人、登记时间、IP地址）

#### **业务规则**

1. 信访编号一旦生成不可修改。
2. 来访人数 > 5人时，系统自动在信访件上标记"集体访"标签，并在待办管理中提升优先级。
3. 信访人身份证号、电话号码、住址在数据库中AES-256加密存储。
4. 登记提交后信访件状态为"待分类处置"。
5. 信访人的敏感信息在前端展示时，非信访管理员角色默认脱敏（身份证号显示前6位+后4位，电话显示前3位+后4位）。

<a id="313-判重管理pt-03"></a>
### **3.1.3 判重管理（PT-03）**

#### **业务背景**

重复信访是信访工作中的常见现象——同一人就同一事项多次反映，或者不同人（如夫妻、同事）就同一事项分别反映。如果不加识别将重复件当作新件独立处理，会导致多头办理、意见矛盾、资源浪费。《信访工作条例》鼓励"减少重复信访"，判重是实现源头治理的关键环节。

传统人工判重依赖信访工作人员记忆和历史登记本翻阅，准确率低、效率差。本系统通过三级判重规则自动比对，仅在复杂关联场景需要人工确认。

#### **功能详述**

**（1）判重触发时机**

- 信访件登记提交时（PT-02提交按钮点击后）自动触发
- 网上信访用户端提交信访时（QZ-01提交后）自动触发
- 支持人工手动触发判重（在信访件详情页"重新判重"按钮）

**（2）三级判重规则**

| 级别 | 规则名称 | 匹配条件 | 权重 | 说明 |
|------|----------|----------|------|------|
| 精确匹配 | 身份完全一致 | 信访人姓名 + 身份证号完全一致 | 高 | 同一人再次信访，确认为重复件的置信度极高 |
| 模糊匹配 | 身份高度相似 | 信访人姓名 + 联系电话 + 诉求关键字相似度 ≥ 80% | 中 | 可能是同一人换了证件或信息录入有差异 |
| 关联匹配 | 员工编号一致 | 员工编号完全一致 | 高 | 本企业员工通过不同渠道反映，通过HR系统员工编号关联 |

**相似度计算逻辑**：
- 姓名相似度：基于编辑距离（Levenshtein Distance）算法，考虑"王芳"与"王方"（同音字）、"张小明"与"张晓明"（漏字）等常见差异
- 诉求关键字相似度：基于TF-IDF提取关键词向量，计算余弦相似度（Cosine Similarity）
- 综合相似度 = 姓名相似度 × 0.4 + 关键字相似度 × 0.6
- 综合相似度 ≥ 80% 视为模糊匹配命中

**（3）判重结果分类与处理**

| 判重结果 | 条件 | 系统行为 |
|----------|------|----------|
| 确认为重复 | 精确匹配或关联匹配命中 | 弹窗展示匹配到的历史件信息，由登记员确认为重复件 |
| 疑似重复 | 模糊匹配命中，综合相似度80%-95% | 弹窗展示相似历史件列表，由登记员判断 |
| 确认为新件 | 无匹配或综合相似度 < 80% | 静默通过，信访件正常进入下一环节 |
| 暂存待确认 | 登记员无法立即判断 | 信访件保存为"草稿"状态，暂不进入流转 |

**（4）判重提醒弹窗设计**

弹窗内容：
- 顶部提示："系统检测到X条疑似重复的历史信访件，请确认"
- 历史件列表（表格形式）：

| 列名 | 说明 |
|------|------|
| 信访编号 | 历史信访件编号，点击可查看详情 |
| 信访人 | 脱敏展示 |
| 登记日期 | 历史件登记日期 |
| 诉求摘要 | 前50字 |
| 当前状态 | 办理中/已办结/已归档 |
| 匹配规则 | 说明通过哪条规则匹配（精确/模糊/关联） |
| 相似度 | 仅模糊匹配时显示百分比 |

- 操作按钮：
  - "确认为重复件"：选择一条历史件作为原件的关联对象，当前件标记为"重复件"，与原件建立关联关系，不独立进入流转
  - "确认为新件"：忽略判重提醒，信访件正常进入待分类处置
  - "暂存"：保存为草稿，后续从草稿箱中继续处理

**（5）重复件处理**

确认为重复的信访件：
- 不生成独立的办理任务
- 与原信访件建立"关联件"关系（在数据库中以关联表记录，支持1:N）
- 在原信访件的详情页"关联件"标签中列出
- 重复件的诉求内容合并到原件的办理参考信息中
- 重复件不独立归档，随原件一同归档

**（6）判重日志**

所有判重操作记录日志：
- 判重时间
- 操作人
- 判重结果（重复/新件/暂存）
- 匹配到的历史件编号（如有）
- 操作人选择的确认动作

#### **业务规则**

1. 判重功能仅用于辅助判断，最终决策权归登记员（信访管理员）。
2. 重复件关联一旦建立不可自动解除，如需解除需走数据修正流程（需管理员权限）。
3. 判重响应时间 ≤ 3秒（含10万条历史记录的全表检索和文本相似度计算）。
4. 历史信访件中被标记为"已删除"（逻辑删除）的不参与判重比对。

<a id="314-分类处置pt-04"></a>
### **3.1.4 分类处置（PT-04）**

#### **业务背景**

登记完成且确认非重复后，信访管理员需对信访件进行分类处置——判断这件信访该由谁来办、以什么方式办。这是信访工作流程中的关键决策节点。《信访事项网上办理工作规程》第六条至第十一条对信访事项的转送、交办、受理等环节的处置方式作出了详细规定。

#### **功能详述**

**（1）五种处置方式**

| 处置方式 | 适用场景 | 后续流程 | 时限要求 |
|----------|----------|----------|----------|
| 直接答复 | 咨询类、政策解释类简单诉求，信访科可直接答复 | 填写答复内容 → 办结归档 | 登记后5个工作日内答复 |
| 转办 | 需同级业务部门办理的诉求 | 选择主办部门 → 转出 → 签收 → 办理 → 反馈 | 登记后3个工作日内转出 |
| 交办 | 需下级单位调查处理的诉求 | 选择下级单位 → 交办 → 签收 → 办理 → 反馈 | 登记后3个工作日内交办 |
| 呈报 | 重大疑难事项，需领导批示 | 填写呈报签 → 呈报领导 → 等待批示 → 按批示处置 | 登记后2个工作日内呈报 |
| 存查 | 无实质诉求（感谢信、已解答咨询等） | 直接归档，不进入流转 | 即时归档 |

**（2）智能推荐**

系统基于可配置规则表智能推荐处置方式和主办部门：

推荐逻辑（规则表可由信访管理员维护）：
- 若诉求类别为"政策咨询"类，推荐"直接答复"
- 若诉求内容不含实质诉求关键词（如"感谢""表扬"），推荐"存查"
- 若涉及单位为单个二级单位，推荐"交办"至该单位的上级
- 若涉及单位为职能部门，推荐"转办"至该部门
- 若诉求类别为"重大疑难"或涉及单位 ≥ 3个，推荐"呈报"
- 若诉求涉及"群体性""安全"等关键词且人数 > 5，推荐"呈报"

推荐结果以标签形式展示在处置选项中（如"转办（推荐）"），信访管理员可一键采用或手动修改。

**（3）拟办意见**

无论选择何种处置方式，信访管理员必须填写"拟办意见"，包含以下三项内容：

| 项目 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 分流理由 | 文本(500) | 是 | 为什么选择这种处置方式 |
| 办理要求 | 文本(1000) | 是 | 对承办单位的具体要求（如需要调查哪些方面、重点关注什么） |
| 注意事项 | 文本(500) | 否 | 特殊提醒事项（如"信访人情绪激动，注意沟通方式"） |

拟办意见在后续流转中作为只读信息展示，承办单位签收时可查看。

**（4）直接答复操作**

选择"直接答复"后：
1. 填写答复内容（文本，≤5000字符）
2. 可选上传答复函附件（仅pdf格式）
3. 提交后信访件状态变为"已答复"
4. 系统自动生成办结报告（PT-08），进入归档流程

**（5）存查操作**

选择"存查"后：
1. 系统弹出二次确认："确认将此信访件标记为存查？存查后信访件直接归档，不进入流转。"
2. 确认后信访件状态变为"已存查"，直接归档
3. 存查的信访件仍可在查询中检索查看

#### **业务规则**

1. 分类处置为信访管理员的专属权限，其他角色不可操作。
2. 处置方式一经提交不可直接修改，如需变更需走"撤回重分"流程（由信访管理员发起，记录变更原因）。
3. 拟办意见三项内容中"分流理由"和"办理要求"为必填，未填写时提交按钮灰显。
4. 智能推荐结果仅供参考，信访管理员有完全决策权。

<a id="315-转交办管理pt-05"></a>
### **3.1.5 转交办管理（PT-05）**

#### **业务背景**

转交办是信访事项流转的核心环节。《信访事项网上办理工作规程》第六条明确要求：对属于下级机关职权范围的信访事项，应通过系统转送、交办。本模块实现交办（上级→下级）、转办（平级→部门）两种派单模式，以及转办下的协办（1对多）机制。

#### **功能详述**

**（1）交办（上级→下级）**

**交办发起**：
- 仅可选当前单位的直接下级单位作为主办单位（通过组织树展示，非下级单位不可选）
- 设定办理时限：默认30个自然日（可手动调整，范围5-90天）
- 可要求下级单位提交办理情况报告、答复意见书等反馈材料（多选配置）

**交办后流转**：
- 交办方仍可实时查看办理进度（含办理过程记录和当前状态）
- 交办方可对办理滞后的交办件发起督办（PT-07）
- 交办方可在交办件办理过程中补充材料或追加办理要求

**退回申请机制**：
- 下级单位若认为该信访事项不属于自身职责范围，不得直接退回
- 必须提交退回申请（填写退回理由及建议转办方向，≤2000字符）
- 退回申请经交办方信访管理员审批：同意退回（重新选择主办单位）或驳回退回申请（继续由该单位办理并附说明）
- 退回申请审批期间，办理时限暂停计算

**（2）转办（平级→部门）**

**转办发起**：
- 仅可选平级职能部门作为主办部门
- 可同时指定协办部门（0至多个），对每个协办部门需说明协助事项
- 对主办部门设定办理时限（默认30天）

**主办与协办关系**：
- 主办部门负责牵头调查、汇总协办部门意见、形成最终答复意见
- 协办部门在收到协办请求后，在指定时限内（默认10天）向主办部门提交协办回复
- 主办部门对办理结果负总责，协办部门承担配合责任
- 协办部门的配合情况纳入其配合度统计（用于绩效考核参考）

**动态协办管理**：
- 办理过程中，主办部门可向信访科申请增加协办部门（填写增加理由和协助事项）
- 信访科审批同意后，新增协办部门收到协办任务通知
- 协办部门在办理过程中也可申请退出（如确与本部门无关），需主办部门确认和信访科审批

**（3）时限管理**

**时限计算规则**：
- 承办单位签收后开始计算办理时限（签收日期为第1天）
- 时限按自然日计算（含周末和节假日），但遇法定节假日可顺延
- 若需延期，承办单位可提交延期申请（说明理由及延期天数，延期天数 ≤ 30天）
- 延期申请经信访科审批后，办理时限顺延
- 呈报领导批示期间，原办理时限暂停计算（从呈报发起日至批示回传日不计入时限）
- 退回重办后，重新设定办理时限（从重新签收日算起）

**时限预警节点**：

| 节点 | 触发条件 | 系统行为 |
|------|----------|----------|
| 时限过半 | 已过办理时限的50%且无过程记录 | 自动向承办人推送提醒消息 |
| 临期预警（7天） | 剩余时限 ≤ 7天 | 待办列表颜色变橙，推送临期提醒 |
| 临期预警（3天） | 剩余时限 ≤ 3天 | 待办列表颜色变红，推送紧急提醒 |
| 临期预警（1天） | 剩余时限 = 1天 | 推送紧急提醒，抄送承办人上级 |
| 超期 | 剩余时限 = 0 | 自动标记超期，纳入督办台账备查，推送超期通知 |

**（4）转交办操作界面交互**

转交办页面分为三个区域：
- **左侧**：信访件信息摘要（信访人脱敏信息、诉求类别、诉求内容摘要、拟办意见）——只读展示
- **中间**：转交办配置区（选择主办方、协办方、设定时限、填写转交办要求）
- **右侧**：组织架构树（按三级展开，支持搜索定位，已选单位高亮显示）

提交时系统自动生成交办函/转办函（含信访件基本信息、拟办意见、办理要求、时限说明），可在提交前预览。

#### **业务规则**

1. 交办只能向下级单位发起；转办只能向平级部门发起。系统在组织架构树中自动过滤不合规的目标单位。
2. 每个信访件同时只能有一个主办单位/部门，协办部门可以有多个。
3. 退回申请在审批期间办理时限暂停，审批不通过则恢复计时。
4. 交办函和转办函自动生成后保存在报告函件台账中（TZ-01），可后续查阅打印。

<a id="316-办理与反馈pt-06"></a>
### **3.1.6 办理与反馈（PT-06）**

#### **业务背景**

承办单位签收后进入实质办理阶段。此模块记录承办单位的办理过程、协办协调情况和最终反馈。《信访事项网上办理工作规程》第十六条明确了办理基本流程：联系/约见信访人 → 核实情况/调查 → 重大复杂可听证 → 作出处理出具《信访事项处理意见书》 → 落实处理意见。

#### **功能详述**

**（1）办理过程记录**

承办单位工作人员可分阶段多次添加办理过程记录，形成可视化时间线：

每次过程记录包含：
- 记录类型：调查取证/沟通协调/采取措施/政策研究/其他
- 记录时间：自动获取当前时间
- 记录内容：文本（≤2000字符）
- 附件：可上传相关佐证材料（≤5个文件，单文件≤50MB）

过程记录按时间倒序排列，以时间线（Timeline）形式展示，最新的记录在最上方，每条记录显示记录人姓名和记录时间。

**（2）协办协同**

当信访件涉及协办部门时，办理页面展示协办协同区域：

- **发起协办请求**：主办部门填写协办事项说明和回复时限，一键发送至所有协办部门
- **协办回复查看**：各协办部门的回复以列表展示（部门名称、回复时间、回复内容摘要、附件），点击展开查看详情
- **协办超期提醒**：协办部门超期未回复（超过指定回复时限3天），系统自动发起协办催办
- **协办意见汇总**：主办部门可一键将各协办部门的回复内容汇总到最终反馈中

**（3）最终反馈**

承办单位需提交最终办理结果，包含：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 办理结果 | 富文本(≤10000字符) | 是 | 详细的调查情况和处理结果 |
| 答复信访人情况 | 文本(≤2000字符) | 是 | 以何种方式、何时答复信访人，信访人的反馈 |
| 是否息诉罢访 | 下拉 | 是 | 是/否/部分接受 |
| 答复意见书 | 附件(PDF) | 是 | 《信访事项处理意见书》扫描件或电子版，必须上传 |
| 证据附件 | 附件(多文件) | 否 | 调查证据、协议书、照片等佐证材料 |

**（4）两级审核**

反馈提交后须经两级审核：

**第一级：内部审核**（承办单位内部负责人）
- 审核人：承办单位指定的审核人（由系统根据单位配置自动路由或手动选择）
- 审核结果：通过（进入第二级审核）/ 退回（附退回原因，退回后承办人修改重新提交）
- 审核时限：反馈提交后3个工作日内完成

**第二级：信访科审核**（信访管理员）
- 审核人：信访管理员
- 审核结果：通过（可进入办结归档）/ 退回（附退回原因）
- 退回原因分类：事实不清（需补充调查）、证据不足（需补充附件）、答复不规范（需修改答复意见书格式）、其他

退回不计入办理时限（退回修改期间的时限暂停计算），重新提交后恢复计时。

#### **业务规则**

1. 最终反馈必须至少上传1个附件（答复意见书），否则提交按钮灰显。
2. 两级审核中任一审核不通过，反馈回到承办人"待反馈"列表中，并附带退回原因。
3. 办理过程记录不可删除和修改（保证留痕），但可追加补充说明。
4. 协办部门的回复内容对信访人不可见（内部工作信息）。

<a id="317-督办管理pt-07"></a>
### **3.1.7 督办管理（PT-07）**

#### **业务背景**

《信访事项网上办理工作规程》明确要求：对交办、转送的信访事项，交办、转送机关要通过信访信息系统及时检查受理、办理情况，发现存在违反信访工作规定受理、办理信访事项等情形的，应当及时督办。督办是保障信访事项按时按质办结的刚性约束手段。

#### **功能详述**

**（1）四种督办触发条件**

| 编号 | 触发条件 | 触发方式 | 说明 |
|------|----------|----------|------|
| DB-01 | 办理时限过半但无任何过程记录 | 系统自动检测 + 人工确认 | 承办单位签收后长时间无实质动作，系统检测到后生成待确认督办建议，信访管理员可确认发起或忽略 |
| DB-02 | 信访人再次反映或上级过问 | 手动发起 | 信访管理员接到信访人再次来电/来访或上级部门询问后手动发起督办 |
| DB-03 | 反馈被退回超过5个工作日未重新提交 | 系统自动发起 | 退回后承办单位修改拖延，系统自动生成督办单 |
| DB-04 | 协办部门超3天未回复协办请求 | 系统自动发起 | 自动向协办部门发起协办催办（轻量级督办） |

**（2）督办单内容**

督办单包含：

| 字段 | 说明 |
|------|------|
| 督办编号 | DB + 年份 + 4位流水号（如DB20260001） |
| 关联信访编号 | 被督办的信访件编号 |
| 被督办方 | 承办单位/部门名称 |
| 督办原因 | 从四种触发条件中选择，可补充说明 |
| 督办要求 | 具体整改或推进要求（文本，≤2000字符） |
| 限定反馈日期 | 默认7天，可手动调整 |
| 发起人 | 自动获取当前用户 |
| 发起时间 | 自动获取当前时间 |

**（3）督办反馈**

被督办方需在限定反馈日期内提交督办反馈：
- 反馈内容：说明当前进展或整改措施（文本，≤3000字符）
- 反馈附件：可上传佐证材料
- 提交后督办单状态变为"已反馈"，等待督办发起方确认
- 督办发起方确认后可关闭督办（状态变为"已闭环"）

**（4）督办台账**

信访管理员有独立的督办台账视图，展示以下信息：
- 所有督办的列表（含督办编号、关联信访编号、被督办方、发起时间、限定日期、状态）
- 状态筛选：全部/待反馈/已反馈/已闭环
- 超期未反馈的督办公示顶（被督办方超期未反馈的督办标红显示）
- 点击督办编号查看完整督办记录（督办单→反馈→闭环的全过程时间线）

#### **业务规则**

1. 督办单一旦发出不可撤回，但可追加说明。
2. 督办反馈提交后，被督办方不可再修改，如需修改需经督办发起方退回。
3. 督办次数与信访件关联，在统计分析模块中统计各单位的被督办次数。

<a id="318-办结归档pt-08"></a>
### **3.1.8 办结归档（PT-08）**

#### **业务背景**

信访件经过完整流程后进入办结归档环节，标志着该信访事项处理完毕。《信访工作条例》要求办理过程和结果可追溯、可查询。归档是将信访件从活跃流转状态转为档案保管状态的标准化操作。

#### **功能详述**

**（1）办结条件**

以下四个条件全部满足，信访件才可进入办结流程：

1. **反馈经两级审核通过**：内部审核和信访科审核均为"通过"
2. **办理程序完整**：各环节均有记录（登记→分类处置→交办/转办→签收→办理过程→反馈→审核），无跳过环节
3. **必要附件齐全**：至少包含《信访事项处理意见书》
4. **信访管理员确认办结**：信访管理员在审核通过后点击"确认办结"

**（2）办结报告自动生成**

系统根据信访件的全流程数据自动生成办结报告，包含：
- 事项摘要（信访编号、信访人脱敏信息、诉求类别、诉求概述）
- 办理时间线（登记日期→分类处置日期→交办/转办日期→签收日期→反馈提交日期→审核通过日期→办结日期）
- 办理过程概要（历次过程记录的摘要）
- 最终办理结果
- 是否息诉罢访
- 附件清单（文件名、类型、大小、上传时间）
- 审核记录（内部审核人/时间/结果、信访科审核人/时间/结果）

办结报告支持导出为PDF格式，可打印归档。

**（3）归档操作**

信访管理员确认办结后：
- 信访件状态变为"已归档"
- 归档后默认为非活跃状态，不再出现在待办列表中
- 设定保管期限：1年/3年/5年/10年/永久（下拉选择，默认5年）
- 归档后不可修改（如确需更正，发起"归档后更正"流程，需管理员权限）
- 保管到期前3个月，系统自动向信访管理员推送提醒"以下信访档案即将到期，请确认是否续期或销毁"
- 支持将已归档信访件的案卷信息推送至企业档案系统（通过接口，详见外部接口需求）

**（4）归档后查询**

已归档信访件仍可在信访档案台账（TZ-01）中检索：
- 按信访编号、信访人、时间范围、诉求类别等条件查询
- 查看详情（只读，不可编辑）
- 导出办结报告
- 关联件查看（查看与该件关联的重复件）

#### **业务规则**

1. 已归档信访件不可修改，如确需更正，需发起"归档后更正"申请，经信访科负责人审批。
2. 归档时系统自动检查保管期限，信访管理员可手动调整。
3. 归档操作记录操作日志（操作人、时间、保管期限）。

<a id="319-查询导出pt-09"></a>
### **3.1.9 查询导出（PT-09）**

#### **功能详述**

**（1）多维组合查询**

支持以下查询条件组合：

| 查询条件 | 类型 | 说明 |
|----------|------|------|
| 信访编号 | 精确/模糊 | 支持前缀匹配 |
| 信访人姓名 | 模糊 | 脱敏存储的字段按实际值匹配 |
| 信访人电话 | 精确 | 仅信访管理员可见此条件 |
| 涉及单位 | 多选(组织树) | 选中某单位，查询包含该单位的所有信访件 |
| 诉求类别 | 级联(三级) | 选择某级后自动带入其下级 |
| 办理状态 | 多选 | 待分类/办理中/已反馈/已办结/已归档/存查 |
| 来源渠道 | 多选 | 来信/来访/来电/网上信访/上级转送/领导批示 |
| 登记日期段 | 日期范围 | 起止日期 |
| 办结日期段 | 日期范围 | — |
| 是否集体访 | 下拉 | 是/否/全部 |
| 关键字 | 文本 | 在诉求内容中全文检索（通过Elasticsearch） |

查询条件可保存为"常用查询"（命名后保存），下次登录后从"我的常用查询"中一键加载。

**（2）查询结果列表**

查询结果以列表形式展示，支持：
- 列自定义显示（勾选要显示的列）
- 按任意列排序
- 点击信访编号进入详情页
- 列表底部显示总条数和当前页码

**（3）导出功能**

查询结果支持导出为Excel格式：
- 导出前可选择包含的字段（默认全部，但敏感字段需勾选确认）
- 默认敏感字段（身份证号、电话全号）脱敏导出
- 如需导出含完整敏感字段，需走特批流程（填写导出用途+审批）
- 导出数据量 > 5000条时，系统异步生成并提示"正在生成导出文件，完成后将通知您"
- 导出记录日志（导出人、时间、导出条件、导出条数、是否含敏感字段）

#### **业务规则**

1. 信访管理员可全量查询 + 含敏感字段导出（需审批）。
2. 业务办理人仅可查询本部门办理的信访件，且默认脱敏显示。
3. 查询条件中涉及敏感字段（电话、身份证号）的仅信访管理员可见。

<a id="3110-催办推送pt-10"></a>
### **3.1.10 催办推送（PT-10）**

#### **业务背景**

催办推送是时限管理的自动化执行环节。在传统工作模式下，催办依赖人工跟踪——信访工作人员需要定期翻阅台账、查看日历，手动打电话或发邮件提醒。这种方式效率低、容易遗漏，且催办是否发出、是否被接收缺乏系统化记录。

#### **功能详述**

**（1）自动催办触发场景**

| 触发方式 | 业务场景 | 推送对象 | 推送时机 |
|----------|----------|----------|----------|
| 自动 | 新转交办任务到达 | 主办单位/部门承办人 | 任务生成即时推送 |
| 自动 | 办理时限过半（≥50%） | 主办单位承办人 | 达到总时限50%时推送 |
| 自动 | 办理时限临期预警 | 主办单位承办人 | 剩余7天、3天、1天各推送一次 |
| 自动 | 办理超期 | 承办人 + 单位负责人 + 信访科 | 超期当日及之后每3天推送一次 |
| 自动 | 协办超期未回复 | 协办部门承办人 | 协办时限超期后每3天推送 |
| 手动 | 信访科发起督办 | 被督办单位承办人及负责人 | 督办发起即时推送 |
| 手动 | 信访管理员手动催办 | 指定接收人 | 即时推送 |

**（2）推送内容**

每条催办推送包含以下信息：
- 推送类型标识：新任务/临期提醒/超期警告/督办通知/协办催办
- 事项编号及信访人姓名（脱敏处理，如"张**"）
- 诉求类别及简要描述（不超过50字）
- 当前环节说明
- 剩余时限或超期天数
- 处理入口链接（点击可直接跳转至PC端办理页面或移动端处理页面）

**（3）催办记录**

每次催办推送需记录：
- 推送时间
- 推送对象（用户ID和姓名）
- 推送方式（掌上新油/短信备选）
- 推送内容摘要
- 是否已读（掌上新油平台回执）
- 是否点击处理（从推送入口进入办理页面的记录）

在信访件详情页的"催办记录"标签中，可查看该件的完整催办历史时间线。

#### **业务规则**

1. 自动催办由系统定时任务触发（每30分钟扫描一次），满足条件即推送。
2. 同一事项同一时间节点不重复推送（如剩余3天时已推送，剩余2天时不再推送，直至剩余1天时再次推送）。
3. 催办推送不可撤回，但可在催办记录中追加备注。

<a id="32-专项工作子系统zx"></a>
## **3.2 专项工作子系统（ZX）**

专项工作子系统服务于信访维稳的延伸业务场景，包含6个模块。

### **3.2.1 不稳定因素排查（ZX-01）**

#### **业务背景**

不稳定因素排查是信访维稳工作的"侦察兵"——各单位定期对可能引发信访或群体性事件的隐患进行摸排登记。传统排查依赖纸质表格下发、手工填写、人工汇总，耗时长、易遗漏、难跟踪。本模块将排查工作线上化，并与信访办理模块和统计分析模块实现联动。

#### **功能详述**

**（1）排查登记**

排查编号自动生成（格式：PC + 年份 + 4位流水号，如PC20260001）。

排查登记字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 排查日期 | 日期 | 是 | 开展排查的日期 |
| 排查单位 | 下拉(组织树) | 是 | 自动带入当前用户所属单位，可修改 |
| 排查人员 | 文本(50) | 是 | 实际执行排查的人员姓名 |
| 隐患类型 | 级联(三级) | 是 | 如：劳动关系→薪酬争议→加班费纠纷 |
| 涉及人员 | 文本(200) | 是 | 涉及的人员姓名及基本信息 |
| 涉及人数 | 整数 | 否 | 若为群体性隐患，填写预估涉及人数 |
| 隐患描述 | 文本(≤2000字符) | 是 | 详细描述隐患的具体情况 |
| 风险等级 | 下拉 | 是 | 高/中/低，根据影响范围和紧急程度判定 |
| 稳控责任人 | 下拉(用户选择) | 是 | 负责稳控的具体人员 |
| 化解措施 | 文本(≤2000字符) | 是 | 计划采取的化解措施 |
| 化解时限 | 日期 | 条件必填 | 高/中风险必填，低风险选填 |
| 当前状态 | 下拉 | 是 | 未化解/化解中/已化解/反弹 |

**（2）批量导入导出**

- 支持按信访科提供的Excel模板导入（≤500条/次），系统自动校验格式和必填字段
- 校验不通过的行生成错误报告（标注行号、错误字段、错误原因），不影响正确行的导入
- 导出排查结果用于上报（按查询条件筛选后导出Excel）

**（3）多维检索**

支持按以下条件组合查询：
- 排查日期段
- 排查单位
- 隐患类型
- 风险等级
- 当前状态

**（4）预警联动**

以下排查项自动在统计分析模块中产生预警：
- 高风险且未化解
- 中风险且化解时限已超期
- 当前状态为"反弹"

### **3.2.2 重点事项化解（ZX-02）**

#### **功能详述**

**（1）化解任务来源**

- 从信访办理模块将重点信访事项标记为"纳入化解"（在信访件详情页操作），自动创建化解任务
- 手动新建化解任务（适用于非信访渠道来源的重点事项）

**（2）化解台账字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 化解编号 | 自动生成 | — | HJ + 年份 + 4位流水号 |
| 事项名称 | 文本(200) | 是 | 化解事项的简要名称 |
| 关联信访件 | 多选关联 | 否 | 关联的信访件编号（1:N） |
| 包案领导 | 下拉(用户选择) | 是 | 负责包案的领导干部 |
| 责任单位 | 多选(组织树) | 是 | 需指定一个主办单位 |
| 化解方案 | 文本(≤5000字符) | 是 | 详细的化解方案 |
| 当前进度 | 百分比滑块 | 是 | 0%-100% |
| 预计化解时间 | 日期 | 是 | 计划完成化解的日期 |
| 实际化解时间 | 日期 | 否 | 实际完成的日期 |
| 化解结果 | 下拉 | 否 | 成功/部分/未化解/转入司法 |
| 后续稳控措施 | 文本(≤2000字符) | 否 | 化解后的持续稳控措施 |

**（3）动态跟踪**

支持分阶段添加化解动态记录，形成时间线。每次动态记录：时间、内容（≤2000字符）、附件（可选）。动态记录按时间倒序展示。

### **3.2.3 稳定风险评估（ZX-03）**

#### **功能详述**

**（1）评估对象**

涉及员工切身利益的政策调整、重大项目、机构改革等。

**（2）评估登记**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 评估编号 | 自动生成 | — | PG + 年份 + 4位流水号 |
| 评估事项名称 | 文本(200) | 是 | — |
| 实施单位 | 下拉(组织树) | 是 | — |
| 评估起止时间 | 日期范围 | 是 | — |
| 评估负责人 | 下拉(用户选择) | 是 | — |
| 事项概述 | 文本(≤2000字符) | 是 | 评估事项的背景和主要内容 |

**（3）四维评估（参考中原油田实践经验）**

| 维度 | 说明 | 评分 |
|------|------|------|
| 合法性 | 是否符合法律法规和集团公司政策 | 1-10分 |
| 合理性 | 是否符合员工合理预期、是否公平 | 1-10分 |
| 可行性 | 实施条件是否具备、时机是否成熟 | 1-10分 |
| 可控性 | 风险是否可以管控、有无应急预案 | 1-10分 |

每个维度需说明评分依据和主要风险点。

**风险等级判定**：
- 综合得分 ≥ 32分 → 低风险
- 24分 ≤ 综合得分 ≤ 31分 → 中风险
- 综合得分 < 24分 → 高风险

**（4）审批流程**

评估完成后需经三级审批：
1. 部门审核（评估负责人所在部门负责人）
2. 信访科复核（信访管理员）
3. 领导审批（分管领导）

**（5）关联追溯**

若后续该事项引发信访，可在信访登记时关联至评估记录（在PT-02涉及单位字段旁边提供"关联评估"按钮），形成"评估→信访→化解"的完整链条。

### **3.2.4 领导接访下访（ZX-04）**

#### **功能详述**

**（1）接访计划排班**

信访科可录入领导接访排班表：
- 日期、时段（上午/下午）、接访领导、接访地点、陪同人员
- 支持批量生成（如按月批量排班）
- 系统在接访前1天向接访领导和陪同人员发送日程提醒

**（2）接访登记**

- 现场记录接访情况：来访人信息、诉求内容、领导现场处理意见
- 支持将接访中受理的事项一键转为正式信访登记（自动带入接访领导、地点、日期信息）

**（3）台账统计**

可查询统计：按领导统计接访次数、受理问题数量、办理结果、满意度等。

### **3.2.5 特殊群体管理（ZX-05）**

#### **业务背景**

对缠访闹访老户、精神障碍患者、生活困难等需重点稳控的对象建立信息档案，实现动态管控。

#### **功能详述**

**人员信息档案字段**：
- 基本信息（姓名、身份证号、联系电话、住址）
- 群体类别（缠访老户/精神障碍/生活困难/其他）
- 诉求历史（关联信访件列表，可穿透查看）
- 包保责任人（关联用户）
- 稳控措施
- 动态记录（时间线，每次更新动态含时间、内容、附件）

**权限控制**：
- 按单位隔离：上级可查下级，平级不互查
- 敏感字段（身份证号、住址）需特殊权限查看

### **3.2.6 模板与资料维护（ZX-06）**

#### **功能详述**

**（1）模板上传与分类**

信访科可将排查表、评估表、交办函等标准模板上传至系统。模板按用途分类：排查类、评估类、函件类、统计类。展示模板名称、版本号、更新时间、上传人。

**（2）版本管理**

- 上传新版本时，系统自动保留历史版本（而非覆盖）
- 各单位在业务操作界面（如新增排查记录）下载模板时，默认下载最新版本
- 可查看历史版本并下载（适用于需要参考历史模板的场景）

**（3）导入校验依据**

用户使用"表格导入"功能时，系统依据信访科上传的最新模板校验数据格式和必填字段。模板更新后，导入功能的校验规则同步更新。

<a id="33-统计分析子系统tj"></a>
## **3.3 统计分析子系统（TJ）**

### **3.3.1 访量趋势统计（TJ-01）**

**统计维度**：
- 时间维度：按年/季/月/日统计信访总量
- 渠道维度：按六类渠道分别统计
- 单位维度：按公司整体→各厂处→各作业区三级下钻

**展示方式**：
- 支持柱状图、折线图、饼图三种图表类型切换
- 支持同比（与去年同期比）和环比（与上月比）分析
- 图表与表格双模式展示，可切换

**下钻交互**：
- 点击公司整体的某月访量柱 → 展开各厂处的访量分布
- 点击某厂处 → 展开各作业区的访量分布
- 点击某作业区 → 展示该作业区当月的信访件明细列表

### **3.3.2 人员与诉求类别统计（TJ-02）**

**信访人员构成分析**：
- 按年龄、性别、用工形式、单位分布等统计访量及占比
- 按是否集体访、是否初访统计

**诉求类别分布**：
- 按多级分类（三级）统计，展示各类别占比及变化趋势
- 支持交叉分析：如"某单位 + 某类诉求"的分布情况

### **3.3.3 信访统计表（TJ-03）**

**标准月报/年报自动生成**：
- 核心指标：信访总量、初访量、重复访量、集体访量/人次、办结率、满意率、平均办理天数、超期率
- 统计表模板可定制（信访管理员配置），数据支持导出Excel

**对比分析**：
- 支持多月对比（如1-6月趋势图）
- 支持多单位对比（各厂处的办结率排名）

### **3.3.4 信访风险智能预警（TJ-04）**

#### **业务背景**

参考中原油田大数据分析经验（发现退休待遇问题占信访总量60.8%），说明数据驱动的风险预警能够提前发现趋势性问题。本模块从"人/词/量/群体/事项"五个维度构建预警规则体系。

#### **功能详述**

**（1）五类预警规则**

| 规则编号 | 规则名称 | 触发条件 | 检测频率 | 风险等级 |
|----------|----------|----------|----------|----------|
| YJ-01 | 重复信访预警 | 同一人7天内信访 ≥ 2次 | 实时（每次登记触发） | 高 |
| YJ-02 | 关键词聚集预警 | 同一诉求关键词7天内集中出现 ≥ 5次 | 每日凌晨统计前7天 | 中 |
| YJ-03 | 访量异常预警 | 某单位月访量环比增长 ≥ 50% | 每日凌晨统计上月 | 中 |
| YJ-04 | 重点群体活动预警 | 重点群体人员（ZX-05中建档人员）出现新的信访活动 | 实时（每次登记触发） | 高 |
| YJ-05 | 化解逾期预警 | 高风险事项未在限定时间内启动化解 | 每日检测化解时限 | 高 |

以上预警规则的阈值（如7天、2次、5次、50%）均可由信访管理员在系统配置中调整。

**（2）预警处置闭环**

预警生成后：
1. 系统自动推送预警通知至相关责任人（信访管理员 + 涉及单位负责人）
2. 在统计分析模块的预警列表中显示，状态为"待处置"
3. 处理人点击预警进入处置页面，可选处置动作：
   - "已启动排查"：填写排查情况说明
   - "下发督办"：直接发起督办（联动PT-07）
   - "忽略"：必须填写忽略原因（如"经核实为正常访量波动"）
4. 处置后预警状态变为"已处置"，全流程可追溯

**（3）预警存档与统计**

- 所有预警及处置记录可查询、统计
- 按月生成预警统计报表（预警总数、各类预警占比、处置率、平均处置时间）

### **3.4.2 历史数据迁移（TZ-02）**

#### **业务背景**

油田有20余年的信访数据积累（约10万条记录），以Excel台账和旧系统数据库形式存在。这些数据是判重匹配和趋势分析的基础数据源，必须完整、准确地迁移至新系统。

#### **功能详述**

**（2）数据校验**

- 记录数校验：迁移前后总记录数一致

- 抽样校验：随机抽取5%的数据进行人工比对
- 生成校验报告（含校验项、校验结果、差异说明）

**（3）数据整合**

- 迁移后的数据与现行数据统一存储、统一查询
- 在查询结果中通过"数据来源"字段区分"历史迁移"和"系统录入"

<a id="35-移动端子模块yd"></a>
## **3.5 移动端子模块（YD）**

### **3.5.1 产品定位**

移动端定位为PC端信访管理系统的轻量化延伸工具，以H5页面形式集成至油田移动办公平台"掌上新油"，聚焦"离开工位也能办公"的核心诉求，围绕待办处理、消息接收、现场信息采集三大场景设计。

**目标用户与典型场景**：

| 用户角色 | 典型使用场景 |
|----------|--------------|
| 信访管理员 | 出差、会议间隙审批紧急事项、查看核心指标、接收预警推送 |
| 业务办理人 | 外出调查时查看信访件信息、现场拍照上传佐证材料、碎片时间签收和反馈 |
| 专项工作人员 | 下基层排查时现场录入信息、拍摄隐患照片、更新重点人员动态、一键上报重大隐患 |

系统管理员不单独使用移动端。

### **3.5.2 核心功能**

**（1）待办中心（YD-01）**

- 展示当前用户所有待办任务，按紧急程度排序，超期任务标红置顶
- 每项任务显示：事项编号、信访人（脱敏）、当前环节、剩余时限
- 按任务类型筛选：待签收、待反馈、待审核
- 点击进入详情执行签收、反馈、审核等操作

**（2）消息通知（YD-02）**

- 接收待办提醒、催办通知、超期预警、督办通知、风险预警等推送
- 点击消息直接跳转对应处理页面（"收到即办"）
- 支持查看历史消息列表，消息不重复推送
- 支持免打扰时段设置，紧急事项不受免打扰限制

**（3）信访件处理（YD-03）**

- 查看信访件基本信息（信访人脱敏信息、诉求内容、当前环节）
- 执行签收操作
- 填写反馈内容，支持拍照上传附件（单次限9张，自动压缩至单张≤500KB）
- 申请延期（填写原因和天数）
- 查看办理过程时间线

**（4）领导审批（YD-04）**

- 查看呈报事项摘要和拟办意见
- 快捷选项：同意/转办/退回
- 支持手写签名

**（5）现场信息采集（YD-05）**

- 填写排查信息：隐患类型、涉及人员、风险等级、情况描述
- 拍照上传现场照片（自动压缩）
- 更新重点人员动态
- 一键上报重大隐患（触发即时预警通知）
- 采集数据实时同步至PC端

**（6）信息查询（YD-06）**

- 按编号或信访人姓名快速检索本人经办的信访件
- 查看当日/本月本单位核心指标卡片（访量、办结率、超期数）

**（7）个人设置（YD-07）**

- 消息免打扰时段设置
- 修改密码（对接统一认证系统）
- 退出登录

### **3.5.3 功能边界（移动端明确排除的功能）**

| 排除功能 | 排除原因 |
|----------|----------|
| 信访件完整登记 | 字段多、操作复杂，不适合移动端 |
| 转交办配置 | 需多级选人、配置协办，交互复杂 |

<a id="37-信访群众用户端qz"></a>
## **3.7 信访群众用户端（QZ）**

#### **业务背景**

《信访工作条例》第十八条要求"各级机关、单位应当向社会公布网络信访渠道"，第二十一条要求"方便信访人查询、评价信访事项办理情况"。信访群众用户端是本系统面向油田员工群众的对外服务窗口，其核心设计理念是"让数据多跑路、群众少跑腿"。信访群众通过油田企业微信/"掌上新油"移动平台或电脑浏览器即可访问，不需要安装额外的APP。

**设计理念**：用户端与管理端使用同一套后端API和数据源，但通过权限控制严格隔离——信访群众只能看到自己提交的信访件，只能操作"提交/查询/评价"等有限的用户功能。

### **3.7.1 网上信访大厅（QZ-01）**

**（1）标准式提交**（适用于大多数场景）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 姓名 | 文本(50) | 是 | 已登录用户自动带入 |
| 联系电话 | 文本(11) | 是 | 已登录用户自动带入 |
| 身份证号 | 文本(18) | 否 | 员工可通过员工编号自动关联 |
| 所属单位 | 下拉(组织树) | 是 | 支持搜索 |
| 诉求类别 | 级联(三级) | 是 | 与PT-02口径一致 |
| 涉及单位 | 多选(组织树) | 是 | 选择认为问题相关的单位 |
| 诉求内容 | 文本(≤5000字符) | 是 | 提供"诉求描述指引"：什么人/什么事/什么时间/什么诉求/希望怎样解决 |
| 附件 | 文件上传 | 否 | ≤50MB/个，≤5个 |

**（2）简化式提交**（适用于操作不便的用户，如老年退休职工）

仅需填写姓名、联系电话和一段文字描述（≤2000字符），系统自动尝试提取和归类，后续由信访管理员在PT-02环节补全分类信息。

**（3）提交后反馈**

提交成功后展示：
- 信访编号（格式：XF+年份+月份+流水+WS）
- 预计处理时限（15日内决定是否受理，受理后60日内办结）
- "您可以随时通过本系统查询办理进度"提示
- 同时向信访人手机发送提交成功短信通知

### **3.7.2 我的信访（QZ-02）**

- 展示信访人提交过的所有信访件列表（编号、日期、状态、摘要）
- 状态以颜色标识：绿色（已办结）、黄色（办理中）、蓝色（待受理）
- 点击进入详情页——展示信访件当前状态和办理进度时间线
- **信息过滤**：信访人可见诉求内容和办理进度，但不可见内部审核意见和拟办意见

### **3.7.3 进度查询（QZ-03）**

为未登录用户提供简化的进度查询入口：
- 输入信访编号 + 预留手机号验证 → 查看当前状态和简要进度
- 适用场景：退休职工不记得账号密码，但手上有信访编号

### **3.7.4 结果评价（QZ-04）**

信访件办结后，系统推送评价提醒：
- 评价维度：整体满意度（1-5星）、办理时效满意度、问题解决程度（完全/部分/未解决）、意见建议
- 评价有效期：办结后30天内，超期可补评
- 低分评价（1-2星）自动提醒信访管理员关注
- 评价数据汇总到统计分析子系统

### **3.7.5 个人信息管理（QZ-05）**

- 查看和更新个人基本信息（姓名、电话、所属单位）
- 员工身份信息通过统一认证获取，不可自行修改
- 离退休或外部人员可通过手机验证维护信息

### **3.7.6 政策法规查阅（QZ-06）**

- 政策法规：《信访工作条例》全文、油田信访工作管理办法摘要、信访人权利义务告知书
- 常见问题FAQ：如"信访后多久能收到答复？""对办理结果不满意怎么办？"
- 内容由信访管理员在后台维护
- 减少因信息不对称导致的误解性信访

### **4.2.1 业务约束**

| 流转动作 | 发起方 | 接收方 | 业务约束 |

### **4.2.2 数据权限规则（七条）**

| 编号 | 规则名称 | 具体约定 |
|------|----------|----------|
| DP-01 | 同级隔离 | 平级单位之间数据隔离，仅对协办事项开放必要数据访问 |

| DP-04 | 敏感字段脱敏 | 身份证全号、电话号码等，除信访管理员外，其他角色默认脱敏展示 |
| DP-05 | 协办临时授权 | 协办部门仅可查看所协办事项的详情，协办结束后权限收回 |
| DP-06 | 导出分级 | 导出含敏感字段的数据需特批权限 |
| DP-07 | 特殊群体访问控制 | 特殊群体信息按单位隔离，敏感字段需特殊权限 |

<a id="43-数据导出"></a>
## **4.3 数据导出**

在信访办理、专项工作、台账等所有具有列表的模块，均需提供数据导出功能：
- 导出前可筛选数据、选择导出字段
- 默认敏感字段脱敏，需特批权限方可导出完整信息
- 导出格式支持Excel（.xlsx）和PDF
- 导出数据量 > 5000条时异步生成，完成后通知用户下载
- 导出操作记录日志（导出人、时间、导出条件、条数、是否含敏感字段）

<a id="44-操作日志与审计"></a>
## **4.4 操作日志与审计**

系统须记录所有用户的增删改操作：
- 日志字段：操作时间、操作用户、IP地址、操作模块、操作类型（增/删/改/查/导出）、操作对象（如信访编号）、操作结果（成功/失败）、请求摘要
- 登录日志：登录时间、IP地址、设备信息、登录结果（成功/失败/锁定）
- 接口调用日志：调用时间、调用方、接口名称、请求摘要、响应状态
- 日志保留不少于3年，不可删除和篡改
- 支持按时间、用户、操作类型、操作模块等多维度组合查询
- 支持导出审计报表
- 异常行为自动告警（短时间大量数据导出、非工作时间批量查询、越权访问尝试、账号异常登录）

**组织架构表（sys_org）**：单位编码、单位名称、上级编码、层级（1/2/3）、状态（有效/已失效）

**操作日志表（sys_audit_log）**：日志ID、用户ID、IP地址、模块、操作类型、操作对象、操作结果、请求摘要、创建时间（按月分区）

<a id="54-数据约束与校验"></a>
## **5.4 数据约束与校验**

**必填字段约定**：
- 信访登记：信访人姓名、联系电话、诉求类别、涉及单位、诉求内容为必填
- 分类处置：处置方式、拟办意见（含分流理由和办理要求）为必填
- 转交办：主办单位、办理时限为必填
- 最终反馈：办理结果、答复信访人情况、是否息诉罢访、答复意见书附件为必填

**字段格式约束**：
- 身份证号：18位（17位数字+1位校验位），支持15位旧号
- 手机号码：11位数字，以1开头
- 信访编号：固定格式XF + 8位年月 + 4位流水 + 2位渠道
- 诉求内容：≤10000字符
- 办理结果：≤10000字符

**附件约束**：
- 支持格式：pdf、jpg、jpeg、png、doc、docx、xlsx
- 单文件限制：≤100MB（PC端）/ ≤50MB（移动端和用户端）
- 数量限制：≤10个（PC端）/ ≤9个（移动端）/ ≤5个（用户端）
- 禁止上传可执行文件（.exe、.bat、.sh等）

<a id="55-历史数据迁移"></a>
## **5.5 历史数据迁移**

1. 历史数据来源：历年Excel台账（约10万条记录）及旧系统数据库
2. 迁移方式：Excel批量导入（≤5000条/批），支持分批导入
3. 数据清洗规则由信访办公室确认（如统一日期格式、规范诉求分类映射）
4. 导入时校验字段格式和必填项，校验不通过生成错误报告
5. 迁移后校验：总记录数一致、核心字段无丢失、随机抽样5%人工比对
6. 迁移后的数据与现行数据统一存储、统一查询，标记"历史迁移"来源
7. 迁移完成后生成数据迁移校验报告

<a id="六外部接口需求"></a>
# **六、外部接口需求**

## **6.1 接口需求概述**

本系统需与油田现有信息化体系中的多个系统进行数据交互与功能集成。所有接口需遵循油田信息安全规范，传输过程加密（HTTPS/TLS），接口调用需记录日志。接口优先级分为三级：P0（系统上线前必须完成对接）、P1（可在上线后对接）、P2（备选，本期预留接口）。

### **6.2 统一身份认证系统（P0必备）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田统一身份认证系统（LDAP/AD域或统一认证平台） |
| 交互目的 | 实现用户单点登录（SSO），统一管理用户身份凭证 |
| 交互方式 | OAuth2.0协议，登录时实时验证Token，获取用户基本信息 |
| 同步内容 | 用户账号、姓名、所属单位、工号 |
| 同步频率 | 登录时实时认证，用户基本信息每日同步 |
| 异常处理 | 认证失败时提示用户联系统一认证系统管理员，系统记录失败日志 |
| 性能要求 | 单次认证请求响应 ≤1秒，支持高并发排队处理 |

### **6.3 人力资源管理系统（P0必备）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田人力资源管理系统 |
| 交互目的 | 获取组织机构树和员工基础信息，用于信访人信息关联、承办人选择、组织架构展示 |
| 同步内容 | 组织机构（单位名称、层级关系、单位编码）、员工信息（姓名、工号、所属单位、岗位、联系方式） |
| 同步频率 | 每日凌晨定时同步一次，支持手动触发全量同步 |
| 同步耗时 | 单次同步 ≤10分钟 |
| 数据校验 | 同步后自动比对前后数据差异，生成变更日志；已撤销的单位/离职人员在系统中标记为"已失效"而非删除 |
| 异常处理 | 同步失败自动重试3次，仍失败则告警通知系统管理员 |

### **6.4 油田移动办公平台"掌上新油"（P0必备）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田移动办公平台（承载移动端H5页面和消息推送） |
| 交互目的 | 实现待办消息推送和移动端H5页面嵌入访问 |
| 推送内容 | 新任务通知、时限预警通知、催办通知、督办通知、风险预警通知、评价提醒 |
| 推送规则 | 按用户角色和业务归属精准推送；用户可设置免打扰时段；紧急事项不受免打扰限制 |
| 推送时效 | 业务动作完成后即时推送（≤3秒） |
| 移动端页面 | H5页面在移动办公平台内嵌访问，适配平台内置浏览器 |
| 异常处理 | 推送失败自动记录日志并重试，超过重试次数告警 |

### **6.5 企业文件存储服务（P0必备）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田企业NAS或对象存储服务 |
| 交互目的 | 信访件附件、台账附件、模板文件的统一存储与管理 |
| 文件类型 | doc/docx/xls/xlsx/pdf/jpg/png/mp4/zip等 |
| 文件大小 | 单文件 ≤100MB，单次上传 ≤10个文件 |
| 安全要求 | 文件上传下载传输加密，存储加密，访问需经过权限校验 |
| 生命周期 | 附件与业务记录绑定，业务记录删除或归档时附件同步处理 |

### **6.6 企业档案系统（P1可上线后对接）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田企业数字档案管理系统 |
| 交互目的 | 已办结信访件的电子案卷推送至档案系统，实现统一归档管理 |
| 推送内容 | 案卷封面信息、卷内文件列表、全部附件材料 |
| 推送时机 | 信访件办结归档时手动触发或自动推送 |
| 推送状态 | 记录推送状态（成功/失败/待推送），支持失败重试 |
| 本期限度 | 完成接口预留，实现手动推送；自动推送与双向同步列入后续优化范围 |

### **6.7 短信/邮件网关（P2备选）**

| 接口项 | 需求描述 |
|--------|----------|
| 对接目标 | 油田企业短信网关或邮件系统 |
| 交互目的 | 当移动办公平台不可用或用户未安装时，通过短信或邮件发送紧急通知 |
| 触发条件 | 系统监测到移动办公平台推送连续失败，或用户标记为"仅短信/邮件通知" |
| 本期限度 | 作为备选方案，本期预留接口 |

### **6.8 接口通信与安全规范**

| 规范项 | 要求 |
|--------|------|
| 网络环境 | 所有接口在油田企业内网进行数据交互 |
| 传输协议 | HTTPS/TLS 1.2+加密传输 |
| 认证方式 | Token认证或证书认证 |
| 日志记录 | 调用时间、调用方、接口名称、请求摘要、响应状态，保留不少于3年 |
| 超时设置 | 接口调用超时 ≤30秒，超时后自动中断并记录异常 |
